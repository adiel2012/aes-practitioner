\begin{verbatim}
Need SQL and ACID transactions?
├─ Yes
│  ├─ Traditional app, existing database → RDS
│  ├─ Need extreme performance → Aurora
│  └─ Specific needs: Oracle/SQL Server → RDS with that engine
│
└─ Need NoSQL?
   ├─ Key-value, scale to millions of requests/sec → DynamoDB
   ├─ Document database, MongoDB compatible → DocumentDB
   └─ Graph relationships → Neptune

Need caching?
├─ In-memory cache → ElastiCache
├─ Redis features needed → ElastiCache for Redis
└─ Simple caching → ElastiCache for Memcached

Need data warehouse/analytics?
├─ OLAP, BI, big data → Redshift
├─ Query data in S3 → Athena
└─ Real-time analytics → Kinesis Data Analytics

Time-series data?
└─ IoT, metrics, logs → Timestream

\subsubsection{VPC Components Decision Guide}


\textbf{Public vs. Private Subnet:}
\begin{itemize}
  \item \textbf{Public Subnet}: Has route to Internet Gateway (0.0.0.0/0 → IGW)
  \item \textbf{Private Subnet}: No direct internet access; uses NAT for outbound
\end{itemize}


\textbf{NAT Gateway vs. NAT Instance:}
\begin{itemize}
  \item \textbf{NAT Gateway}: AWS managed, highly available, easier, more expensive
  \item \textbf{NAT Instance}: EC2 instance you manage, cheaper, less reliable
\end{itemize}


\textbf{VPC Peering vs. Transit Gateway:}
\begin{itemize}
  \item \textbf{VPC Peering}: Connect two VPCs directly, non-transitive
  \item \textbf{Transit Gateway}: Hub-and-spoke model, simplifies multiple VPC connections
\end{itemize}


\textbf{VPN vs. Direct Connect:}
\begin{itemize}
  \item \textbf{VPN}: Quick setup (hours), encrypted, over internet, variable performance
  \item \textbf{Direct Connect}: Dedicated link, consistent performance, expensive, takes weeks
\end{itemize}


\textbf{VPC Endpoint Types:}
\begin{itemize}
  \item \textbf{Interface Endpoint}: ENI with private IP, uses PrivateLink
  \item \textbf{Gateway Endpoint}: Route table entry, for S3 and DynamoDB only
\end{itemize}

\subsubsection{Migration Decision Tree}


\begin{verbatim}
How much data to migrate?
├─ < 10 TB with good bandwidth → DataSync over network
├─ 10-80 TB or limited bandwidth → Snowcone/Snowball
└─ > 80 TB or very limited bandwidth → Snowball Edge/Snowmobile

What type of data?
├─ Servers/VMs → MGN (Application Migration Service)
├─ Databases → DMS (Database Migration Service)
├─ Files/NAS → DataSync or Snow Family
└─ SFTP workflows → Transfer Family

Need ongoing synchronization?
├─ Yes → DataSync (scheduled) or DMS (continuous)
└─ No → One-time Snow Family transfer

Network bandwidth available?
├─ > 100 Mbps → DataSync
├─ 10-100 Mbps → Consider Snowcone
└─ < 10 Mbps → Snow Family required
\end{verbatim}

\subsubsection{Networking Architecture Decision Tree}


\begin{verbatim}
What do you need to connect?

AWS Resources to Internet
├─ From public subnet → Internet Gateway
├─ From private subnet → NAT Gateway (or NAT Instance)
└─ Private access to AWS services → VPC Endpoints

Multiple VPCs
├─ 2 VPCs → VPC Peering
├─ 3+ VPCs → Transit Gateway
└─ Shared services to many VPCs → PrivateLink

On-Premises to AWS
├─ Quick setup, encrypted → VPN Gateway
├─ Consistent bandwidth, private → Direct Connect
├─ Redundancy needed → Multiple VPN or Direct Connect + VPN
└─ Low latency required → Direct Connect

Content Delivery
├─ Static content globally → CloudFront + S3
├─ Dynamic content → CloudFront + ALB
└─ DNS routing → Route 53

Load Balancing
├─ HTTP/HTTPS → Application Load Balancer (ALB)
├─ TCP/UDP, extreme performance → Network Load Balancer (NLB)
├─ Third-party appliances → Gateway Load Balancer
└─ Legacy (avoid) → Classic Load Balancer

Security
├─ Web application protection → WAF + ALB/CloudFront
├─ DDoS protection → Shield (Standard/Advanced)
├─ Network ACLs → Subnet-level firewall (stateless)
└─ Security Groups → Instance-level firewall (stateful)
\end{verbatim}

\subsubsection{Disaster Recovery Strategy Selector}


\begin{verbatim}
What's your RTO/RPO requirement?

RTO: Minutes, RPO: Seconds
└─ Multi-Site Active/Active
   - Hot standby in multiple regions
   - Highest cost, immediate failover
   - Use: Route 53 health checks, Aurora Global Database
   - Cost: 200\% of single site

RTO: < 1 hour, RPO: Minutes
└─ Warm Standby
   - Scaled-down version running in DR region
   - Can scale up quickly
   - Use: Scaled-down EC2, read replicas promoted
   - Cost: 50-100\% of single site

RTO: Hours, RPO: Minutes to Hours
└─ Pilot Light
   - Core components always running (e.g., databases)
   - Full environment provisioned on failover
   - Use: RDS standby, data replicated to S3
   - Cost: 20-50\% of single site

RTO: Days, RPO: Hours
└─ Backup and Restore
   - Regular backups to S3/Glacier
   - Restore from backup on disaster
   - Use: AWS Backup, S3 cross-region replication
   - Cost: < 10\% of single site (storage only)

Implementation Patterns:
- Database replication → RDS Multi-AZ, Aurora replicas
- Data replication → S3 Cross-Region Replication
- Infrastructure as Code → CloudFormation for quick rebuild
- Traffic management → Route 53 health checks and failover
- Continuous sync → DataSync, Storage Gateway
\end{verbatim}

  \item Real-time file processing
  \item Chatbots and Alexa skills
  \item IoT backend processing
  \item ETL jobs under 15 minutes
\end{itemize}


\textbf{Lambda - NOT Good For:}
\begin{itemize}
  \item Long-running processes (> 15 min)
  \item Applications with consistent traffic (better to use EC2/containers)
  \item Processes requiring local storage > 10 GB
  \item Applications needing GPUs
  \item Legacy applications with complex dependencies
\end{itemize}


\textbf{EC2 - Perfect For:}
\begin{itemize}
  \item Custom applications requiring specific OS
  \item Applications needing GPUs
  \item Lift-and-shift migrations
  \item Consistent workloads (use Reserved Instances)
  \item Applications with licensing restrictions
  \item High-performance computing clusters
  \item Windows-based applications
\end{itemize}


\textbf{EC2 - NOT Good For:}
\begin{itemize}
  \item Simple event-driven functions (use Lambda)
  \item Unpredictable traffic spikes (unless using Auto Scaling)
  \item Microservices that could be containerized
  \item When you want zero administration
\end{itemize}


\textbf{Fargate - Perfect For:}
\begin{itemize}
  \item Microservices architectures
  \item Long-running containers
  \item Batch jobs and scheduled tasks
  \item CI/CD agents
  \item Applications with variable load
  \item When you don't want to manage servers
\end{itemize}


\textbf{Fargate - NOT Good For:}
\begin{itemize}
  \item Consistent high workloads (EC2 with containers cheaper)
  \item GPU workloads (use ECS with EC2)
  \item Windows containers requiring host customization
  \item Extremely latency-sensitive apps (cold starts)
\end{itemize}


\subsubsection{Storage Service Scenarios}


\textbf{S3 Standard - Perfect For:}
\begin{itemize}
  \item Active website content and assets
  \item Frequently accessed data
  \item Content distribution with CloudFront
  \item Data lake hot tier
  \item Mobile and gaming applications
  \item Big data analytics (frequently queried)
\end{itemize}


\textbf{S3 Standard-IA - Perfect For:}
\begin{itemize}
  \item Backups accessed occasionally
  \item Disaster recovery files
  \item Data retained for compliance
  \item Old data accessed few times per month
  \item Long-lived but infrequently accessed
\end{itemize}


\textbf{S3 Glacier - Perfect For:}
\begin{itemize}
  \item Digital preservation
  \item Archive data rarely accessed
  \item Tape replacement
  \item Regulatory archives
  \item Long-term backups
\end{itemize}


\textbf{S3 Glacier - NOT Good For:}
\begin{itemize}
  \item Frequently accessed data (expensive retrievals)
  \item Data needed immediately (retrieval time)
  \item Small files (minimum storage duration charges)
\end{itemize}


\textbf{EFS - Perfect For:}
\begin{itemize}
  \item Web serving and content management
  \item Container persistent storage
  \item Shared data between instances
  \item Big data analytics needing shared access
  \item ML training data accessed by multiple instances
  \item Lift-and-shift applications expecting NFS
\end{itemize}


\textbf{EFS - NOT Good For:}
\begin{itemize}
  \item Single-instance workloads (EBS cheaper)
  \item Windows applications (use FSx for Windows)
  \item Highest IOPS needs (use EBS io2)
  \item Object storage needs (use S3)
\end{itemize}


\subsubsection{Database Service Scenarios}


\textbf{RDS - Perfect For:}
\begin{itemize}
  \item Traditional relational databases
  \item Existing applications using MySQL/PostgreSQL/Oracle/SQL Server
  \item OLTP workloads
  \item When you need Multi-AZ for HA
  \item Read replicas for read-heavy workloads
  \item Lift-and-shift of existing databases
\end{itemize}


\textbf{RDS - NOT Good For:}
\begin{itemize}
  \item Massive scale (consider DynamoDB or Aurora)
  \item Need root access (use EC2 with self-managed DB)
  \item Simple key-value access (use DynamoDB)
  \item Frequent schema changes (consider NoSQL)
\end{itemize}


\textbf{Aurora - Perfect For:}
\begin{itemize}
  \item High-performance OLTP
  \item Applications needing MySQL/PostgreSQL compatibility
  \item Global applications (Global Database)
  \item Variable workloads (Serverless v2)
  \item Read-heavy workloads (15 read replicas)
  \item When you need best RDS performance
\end{itemize}


\textbf{DynamoDB - Perfect For:}
\begin{itemize}
  \item Serverless applications
  \item Mobile backends
  \item Gaming leaderboards
  \item Session stores
  \item Shopping carts
  \item IoT data
  \item Millisecond latency required
  \item Massive scale (millions of requests/sec)
\end{itemize}


\textbf{DynamoDB - NOT Good For:}
\begin{itemize}
  \item Complex queries with joins
  \item Ad-hoc queries and analytics (use RDS or Athena)
  \item ACID transactions across items (use RDS)
  \item When you need SQL
  \item Applications with unpredictable query patterns
\end{itemize}


\textbf{Redshift - Perfect For:}
\begin{itemize}
  \item Business intelligence
  \item OLAP workloads
  \item Complex queries across large datasets
  \item Data warehouse consolidation
  \item Historical data analysis
  \item Petabyte-scale analytics
\end{itemize}


\textbf{Redshift - NOT Good For:}
\begin{itemize}
  \item OLTP workloads (use RDS or DynamoDB)
  \item Real-time data ingestion (use Kinesis)
  \item Small datasets < 100 GB (use RDS or Athena)
  \item High-concurrency workloads (use RDS)
\end{itemize}

\subsection{Service Anti-Patterns (When NOT to Use)}


\subsubsection{Common Mistakes and Anti-Patterns}


\textbf{Lambda Anti-Patterns:}
\begin{itemize}
  \item Using Lambda for long-running processes (use ECS/Fargate)
  \item Storing state in /tmp (use S3 or DynamoDB)
  \item Synchronous invocation chains (use Step Functions)
  \item Not using connection pooling for databases
  \item Ignoring cold start optimization
\end{itemize}


\textbf{S3 Anti-Patterns:}
\begin{itemize}
  \item Using S3 as a database (use DynamoDB or RDS)
  \item Frequent small file updates (use EFS or database)
  \item Sequential file naming causing hot partitions
  \item Not using lifecycle policies for old data
  \item Public buckets without careful review
\end{itemize}


\textbf{EC2 Anti-Patterns:}
\begin{itemize}
  \item Not using Auto Scaling for variable workloads
  \item On-Demand instances for predictable workloads (use RIs)
  \item Over-provisioning instance sizes
  \item Not using EBS snapshots for backups
  \item Forgetting to terminate unused instances
\end{itemize}


\textbf{RDS Anti-Patterns:}
\begin{itemize}
  \item Not using Multi-AZ for production
  \item Single large instance instead of read replicas
  \item Not monitoring storage and IOPS
  \item Storing files/blobs in database (use S3)
  \item Not using parameter groups for tuning
\end{itemize}


\textbf{DynamoDB Anti-Patterns:}
\begin{itemize}
  \item Using it as a relational database with complex joins
  \item Hot partition keys
  \item Storing large items > 400 KB
  \item Not using DynamoDB Streams for replication
  \item Provisioned capacity with unpredictable traffic
\end{itemize}


\textbf{VPC Anti-Patterns:}
\begin{itemize}
  \item Overlapping CIDR blocks preventing peering
  \item Too small CIDR ranges
  \item Not using multiple AZs for HA
  \item Public subnets for databases
  \item Not using VPC endpoints for S3/DynamoDB
\end{itemize}


\textbf{ECS/EKS Anti-Patterns:}
\begin{itemize}
  \item Using Fargate for consistent high-volume (EC2 cheaper)
  \item Not using Application Load Balancer for HTTP services
  \item Ignoring container health checks
  \item Large monolithic containers
  \item Not using ECR for private images
\end{itemize}


\subsection{Common Misconceptions}


\subsubsection{Clearing Up Confusion}


\textbf{S3 Misconceptions:}
\begin{itemize}
  \item "S3 is just for backups" - FALSE: It's for any object storage, including active applications
  \item "S3 is slow" - FALSE: Can handle thousands of requests per second per prefix
  \item "You need to manage S3 servers" - FALSE: Fully serverless and managed
\end{itemize}


\textbf{Lambda Misconceptions:}
\begin{itemize}
  \item "Lambda is always cheaper" - FALSE: Consistent traffic makes EC2/Fargate cheaper
  \item "Lambda scales infinitely" - FALSE: Limited by concurrent execution limit (can be increased)
  \item "Lambda is only for simple functions" - FALSE: Can run complex applications
\end{itemize}


\textbf{VPC Misconceptions:}
\begin{itemize}
  \item "All traffic between AZs is free" - FALSE: \$0.01/GB each direction
  \item "Private subnet means secure" - FALSE: Still need Security Groups and NACLs
  \item "VPC peering is transitive" - FALSE: Must create direct peering between each VPC pair
\end{itemize}


\textbf{RDS Misconceptions:}
\begin{itemize}
  \item "Multi-AZ means read scaling" - FALSE: It's only for HA, use read replicas for scaling
  \item "Automated backups are free" - FALSE: Charged for backup storage beyond DB size
  \item "All databases should use RDS" - FALSE: NoSQL workloads better on DynamoDB
\end{itemize}


\textbf{IAM Misconceptions:}
\begin{itemize}
  \item "Root user should be used daily" - FALSE: Create IAM users and lock down root
  \item "IAM is free so roles don't matter" - FALSE: Proper roles crucial for security
  \item "Users need passwords" - FALSE: Service accounts should use roles, not users
\end{itemize}

\subsubsection{How to Request Service Limit Increases}


\textbf{Method 1: Service Quotas Console} (Preferred)
\begin{enumerate}
  \item Open Service Quotas console
  \item Choose AWS service
  \item Select quota to increase
  \item Request quota increase
  \item Provide use case justification
\end{enumerate}


\textbf{Method 2: Support Case}
\begin{itemize}
  \item For limits not in Service Quotas console
  \item Create support case under "Service Limit Increase"
  \item Provide detailed justification
  \item Include projected usage
\end{itemize}


\textbf{Method 3: Automatic Increases}
\begin{itemize}
  \item Some services auto-increase based on usage (S3, DynamoDB On-Demand)
  \item No action required
\end{itemize}


\textbf{Best Practices:}
\begin{itemize}
  \item Request increases proactively before hitting limits
  \item Monitor usage with CloudWatch and Service Quotas
  \item Set up CloudWatch alarms for approaching limits
  \item Design applications to handle throttling gracefully
  \item Use multiple accounts to multiply limits (via AWS Organizations)
\end{itemize}


\subsubsection{Limits That Affect Architecture}


\textbf{Design Considerations:}

\textbf{EC2 Instance Limits:}
\begin{itemize}
  \item Plan Auto Scaling max size within limits
  \item Use multiple instance types for diversity
  \item Consider Reserved Instances against limits
\end{itemize}


\textbf{Lambda Concurrency:}
\begin{itemize}
  \item Reserve concurrency for critical functions
  \item Use SQS for buffering
  \item Design for throttling (exponential backoff)
\end{itemize}


\textbf{DynamoDB Throughput:}
\begin{itemize}
  \item Use On-Demand for unpredictable traffic
  \item Provision with auto-scaling for predictable
  \item Partition data to avoid hot keys
\end{itemize}


\textbf{S3 Request Rates:}
\begin{itemize}
  \item Use random prefixes for high request rates
  \item CloudFront for read-heavy workloads
  \item Transfer Acceleration for uploads
\end{itemize}


\textbf{VPC Limits:}
\begin{itemize}
  \item Plan IP addressing carefully (CIDR sizing)
  \item Use Transit Gateway for many VPCs
  \item Monitor ENI usage (Lambda in VPC consumes ENIs)
\end{itemize}


\textbf{IAM Limits:}
\begin{itemize}
  \item Use roles instead of users where possible
  \item Group permissions efficiently
  \item Consider AWS SSO for human users
  \item Federation for external identities
\end{itemize}
